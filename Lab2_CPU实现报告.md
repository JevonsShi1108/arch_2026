## Lab2 五级流水 RISC-V CPU 实现报告

### 1 实现目标与约束

Lab2 在 Lab1 已有 64 位五级流水 CPU（IF/ID/EX/MEM/WB）的基础上，增量实现真实的访存能力，需要在既有数据通路之上接入数据总线访问，并保证访存语义的完全正确，具体体现在要支持一整套 Load 指令（`ld/lb/lh/lw/lbu/lhu/lwu`）、Store 指令（`sd/sb/sh/sw`），以及在 Lab2 代码路径中对 `lui` 指令的一致支持。

### 2 总体设计思路

整体设计仍然沿用 `vsrc/src/` 目录下的模块化结构，不做大规模重构，而是围绕各个已有模块做局部增强：`core.sv` 继续作为五级流水的总控单元，只是在原有基础上扩展流水寄存器字段和 stall/flush 的优先级规则；`decode.sv` 增补访存相关指令的译码逻辑与控制位输出，使得各种 Load/Store 与 `lui` 指令都能在译码级被准确区分；`execute.sv` 则在原有 ALU 运算的基础上加入有效地址计算、`rs2` 原值的透传以及访存控制信号的透传；`mem.sv` 从 Lab1 中“简单直通”的形式升级为真正通过 `dbus` 进行读写并具备等待能力的访存级；而 `writeback.sv`、`regfile.sv` 与 `fetch.sv` 等模块，则尽量保持原有的边界接口与时序模型不变。不改变五级流水的基本结构，也不额外引入复杂的新框架，而是在原有的 pipeline register 中增添 Lab2 所必需的字段与控制信息。

### 3 流水寄存器与控制信号扩展

为了支撑 Lab2 的访存需求，`id_ex_t` 与 `ex_mem_t` 两级流水寄存器都增加了若干与访存直接相关的字段，典型包括表示指令类型的 `is_load/is_store` 标志、用于刻画读写宽度的 `mem_size`（可以取 1/2/4/8 字节）、指示是否做无符号扩展的 `load_unsigned`、在 EX 阶段计算得到并供 MEM 级使用的 `mem_addr`，以及对 Store 指令而言需要向后级透传的 `store_data`。其中，`id_ex` 仍然保留了经过前递后的 `op1` 与 `rs2_val`，从而保证 EX 阶段可以在一个时钟周期内同时完成三类工作：一是基于 `op1` 与适当来源（当 `is_imm` 置位时使用 `imm`）的第二操作数进行常规 ALU 运算；二是基于 `op1` 与 `rs2_val` 进行分支比较，避免立即数路径对分支判断带来干扰；三是为访存准备地址与写数据，即通过 `mem_addr = op1 + imm` 计算出有效地址，并将 `store_data` 直接设为 `rs2_val`。这种做法与 Lab1 中“ID 负责前递、EX 负责执行”的分工保持一致。

### 4 译码与执行级改动

#### 4.1 译码（`decode.sv`）

在译码级，Lab2 为整套访存相关指令补全了译码逻辑。对于 `LOAD` 类指令，通过 `funct3` 字段将字节、半字、字与双字访问区分开来，同时还要区分有符号扩展与无符号扩展，从而决定 MEM 级回传数据在 WB 阶段应采用符号扩展还是零扩展的形式。对于 `STORE` 类指令，同样依靠 `funct3` 把 `sb`、`sh`、`sw` 和 `sd` 四种不同写入宽度加以区分，以便在后级正确生成字节选通信号和写入数据。与此同时，`LUI` 指令在译码中会生成 U-type 立即数，并开启对应目的寄存器的写回使能，使其在后续流水中被当作普通写回类指令处理。为了支撑这些新指令类型，译码阶段还补充了 `imm_s` 与 `imm_u` 两种立即数形式，并在不同指令类型下选择合适的立即数格式送入 EX 级。

#### 4.2 执行（`execute.sv`）

在执行级的设计中，没有重写原有 ALU 主体，而是在其周围补充了若干与 Lab2 相关的控制和数据路径。首先，通过引入 `alu_src2 = is_imm ? imm : rs2_val` 这一路径选择，使得 EX 阶段可以根据指令类型自动在立即数和寄存器值之间切换第二操作数；同时，分支比较统一采用 `op1` 与 `rs2_val` 作为比较对，从而把分支判断逻辑和立即数数据通路区分开来，避免因为立即数选择而影响分支的正确性。对于 `lui` 指令，其写回值在 EX 阶段被直接设为 `imm`，以简化后续流水的处理。除此之外，EX 级还负责将前文提到的访存控制信息以及计算得到的有效地址和写数据打入流水寄存器并向 MEM 级透传。

### 5 MEM 级真实访存实现

Lab2 最关键的改动在 `mem.sv` 模块上。在请求生成阶段，对于 Load 指令，模块会在保证 `strobe=0` 的前提下，根据流水寄存器中携带的 `mem_size` 信息发起合适宽度的读请求；对于 Store 指令，则首先依据有效地址的低三位 `addr[2:0]` 计算出经过移位后的字节选通信号 `shifted_strobe` 以及相应对齐后的写数据 `shifted_store_data`，同时保持原始字节地址送往总线，使得底层存储系统能够根据地址与选通信息完成正确的字节写入。在对齐与字节选取阶段，当存储系统返回一个 64 位的数据 beat 时，MEM 级会再次利用地址低三位对数据进行右移，从中提取出真正需要的子字，然后根据具体指令语义执行符号扩展或零扩展：例如，`lb/lh/lw/ld` 等指令采用符号扩展，而 `lbu/lhu/lwu` 等无符号加载指令则采用零扩展。为了配合总线可能存在的等待与握手机制，模块中还设计了“等待与挂起（pending）”逻辑：对于尚未完成的访存事务，会在内部保存当前指令的上下文信息，并一直等待到 `dresp.data_ok` 置位后才输出有效结果，从而避免不同指令之间出现重复提交或越序提交的问题。

### 6 流水停顿与冒险处理

在流水控制方面，Lab2 保持“前递优先”，只针对新的访存行为增加了两类必要的停顿机制。其一是访存等待引起的停顿：当 MEM 级存在尚未完成的访存事务时，流水会冻结前级的推进，同时抑制新的取指请求进入流水，以免在总线仲裁场景下导致数据缓存饥饿或者对同一指令重复发射请求。其二是典型的 load-use 冒险处理：当 ID 级当前指令的数据依赖关系指向 ID/EX 中某条尚未从 MEM 级返回结果的加载指令时，流水会在该位置插入一个气泡，通过最小化的停顿等待数据进入可前递阶段，再继续执行后续指令。整个流水中的转发优先级仍然保持为 `EX > MEM > regfile`，并继续忽略对 `x0` 的伪写回，以避免对零寄存器产生无意义的前递。



在未说明的方面，Lab2 完全沿用了 Lab1 。
